# CRAN submission checklist

Concrete items remaining before `devtools::release()`. Track v0.2
submission here. Reset for future submissions.

## Current status (2026-05-18)

- ✅ `R CMD check --as-cran` locally: 0 errors / 0 warnings / 0 notes
- ✅ GitHub Actions: R-CMD-check passing on ubuntu-latest, macos-latest,
  windows-latest (release + devel + oldrel-1)
- ✅ Test coverage: 37 tests across 5 files, all passing
- ✅ Package size: 0.26 MB tarball (limit 5 MB recommended, 100 MB hard)
- ✅ `cran-comments.md` drafted

## Blocking — must do before submitting

- [ ] **`check_win_devel()`** — submit to win-builder, wait for the
      email confirming clean on R-devel for Windows. CRAN requires this.
      ```r
      devtools::check_win_devel()
      ```
- [ ] **`check_mac_release()` / R-hub** — same idea for macOS.
      ```r
      devtools::check_mac_release()
      # or: rhub::check_for_cran()
      ```
- [ ] **Vignettes knit cleanly with pandoc** — currently CI handles
      this; verify locally that `devtools::build_vignettes()` succeeds.
      Requires pandoc installed (already on the GitHub Actions runners).
- [ ] **Spell check** — catch typos in docs/vignettes before CRAN
      flags them.
      ```r
      devtools::spell_check()
      # whitelist proper nouns in inst/WORDLIST
      ```
- [ ] **URL check** — every URL in DESCRIPTION/man/vignettes must
      resolve (CRAN tolerates redirects but not 404s).
      ```r
      urlchecker::url_check()
      urlchecker::url_update()   # auto-fix redirects
      ```
- [ ] **Examples runtime** — every `@examples` block must run in <5s
      on CRAN. Currently we use `\dontrun{}` only for `run_dashboard()`;
      verify the rest stay under the limit.
- [ ] **`LICENSE` file** — CRAN prefers a separate `LICENSE` (not
      `LICENSE.md`) when the License field references one. For GPL-3
      the standard template is auto-generated by
      `usethis::use_gpl_license(3)`.

## Should do — strongly recommended

- [ ] **`list_sources()` helper** — referenced in
      `inst/dict/vocabularies/sources.yaml` description but never
      implemented. Should return a tibble of all known source keys
      with short_name, type, coverage, doi. ~20 lines, mirrors
      `educabr_cite()`.
- [ ] **Lifecycle badge** — bump from `experimental` to `maturing`
      (`lifecycle::badge("maturing")`) once API is exercised by at
      least one external user. Keep `experimental` for first CRAN
      submission so we have room to break.
- [ ] **`.github/CONTRIBUTING.md`** — clarify "to add a dataset, open
      an issue first" — already mentioned in README, deserves its own
      file for visibility on GitHub's "New PR" prompt.

## Nice to have — defer if time runs short

- [ ] **`pkgdown` deployment via GitHub Actions** — already configured
      (`.github/workflows/pkgdown.yaml`), but the workflow has been
      failing intermittently. Investigate.
- [ ] **Test coverage badge** — `usethis::use_coverage("codecov")`
      and `usethis::use_github_action("test-coverage")`.
- [ ] **Reverse dependency check** — only relevant for revisions, not
      first submission. Use `revdepcheck::revdep_check()` from v0.3
      onward.
- [ ] **CRAN comments per-release** — keep `cran-comments.md` updated
      with what changed since the last submission (CRAN reviewers read
      this).

## After acceptance

- [ ] Tag the release: `git tag -a v0.2.0 -m "First CRAN release"`
- [ ] GitHub release with `gh release create v0.2.0 --notes-from-tag`
- [ ] Zenodo DOI (if integrated): toggle the repo in Zenodo to mint a
      DOI for the GitHub release. Add the DOI to `inst/CITATION`.
- [ ] Bump `Version: 0.2.0.9000` and start v0.3 work on `main`.
